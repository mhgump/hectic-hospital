Hectic Hospital — Merge Strategy

Zero-Conflict Git Workflow for a 6-Hour Hackathon Sprint

The Golden Rule

P1 is the only person who edits PlayState.ts.
That file is where everything meets. One owner = zero conflicts on the most critical file.
Everyone else builds self-contained modules with constructor, update(dt), and dispose().

The 6 Rules
1

Each person creates their OWN files

Never edit the same file as someone else. Each person owns a set of files exclusively. See the ownership table below.

2

P2 and P3 export clean APIs — P1 imports them

P2 and P3 build standalone modules with a clear public interface. They don't wire anything into the game loop. P1 plugs their classes into PlayState.ts.

3

Git branches per person, merge INTO P1's branch

Three branches: p1/gameplay, p2/graphics, p3/physics. P1 merges the others in. Since files don't overlap, merges are conflict-free.

4

Shared types file is written ONCE, then frozen

P1 writes src/hospital/types.ts in the first 15 minutes. After that, nobody changes it without telling the group. If someone needs a new field, they message and P1 adds it.

5

Each person has a standalone way to test

P2 can render rooms without game logic. P3 can test circle collisions without real NPCs. P1 can stub P2/P3's classes with dummy data.

6

Never edit src/main.ts unless absolutely necessary

If someone must touch it (like adding a Babylon side-effect import), they tell the group and one person makes the change.

Git Branch Workflow
main ├── p1/gameplay ← P1 works here (owns PlayState, merges others in) ├── p2/graphics ← P2 works here (never touches PlayState) └── p3/physics ← P3 works here (never touches PlayState)

Creating your branch

# Everyone runs this from main: $ git checkout -b p1/gameplay # or p2/graphics, p3/physics $ git push -u origin p1/gameplay

At each sync point (P2 and P3)

# Commit and push your work: $ git add . $ git commit -m "P2: hospital rooms and waypoints" $ git push

At each sync point (P1 — the integrator)

# Pull in P2 and P3's work: $ git fetch origin $ git merge origin/p2/graphics --no-edit $ git merge origin/p3/physics --no-edit # Wire their new systems into PlayState # Test, commit, push $ git push

After P1 integrates (everyone)

# Rebase onto P1's integrated branch: $ git fetch origin $ git rebase origin/p1/gameplay # Since you own different files, this should be conflict-free
Integration Schedule
0:00 — Kickoff
Everyone together (15 min)
P1 writes types.ts, strips sample game. Everyone agrees on coordinates, creates branches, splits.
0:15 – 1:30
Heads-down building
Each person works on their branch independently. No merges needed yet.
1:30 — Sync 1
First integration
P2 pushes room layout. P3 pushes basic circle collisions. P1 merges both, wires into PlayState.
  • Quick test: Do rooms render? Do circles collide?
1:30 – 3:00
Heads-down building
Everyone rebases from P1's branch, continues building on the integrated codebase.
3:00 — Sync 2
Second integration
P2 pushes character meshes + props. P3 pushes knockback + drag. P1 merges, connects NPCs to layout + physics.
  • Quick test: Do NPCs walk between rooms and bump into each other?
3:00 – 4:30
Heads-down building
Everyone rebases from P1, continues with polish and features.
4:30 — Sync 3
Final integration
P2 pushes visual polish. P3 pushes animations + injury events. P1 merges, connects full pipeline.
  • Quick test: Full patient flow from entry to exit?
5:00 – 6:00
Final hour — everyone on one branch
Merge P1's branch to main. Everyone works on main together. Bug fixes, tuning, demo prep. No more separate branches.

If you follow these rules

You will spend zero minutes on merge conflicts and all your time on building the game. The only coordination needed is a quick Slack/Discord message at sync points: "Pushed, ready for merge."
